# 大統工作助手專案記憶

更新日期：2026-06-29

這是從專案歷史與已確認規則整理出的可攜式記憶，不是原始聊天紀錄。內容刻意排除帳號、token、郵件識別碼、私人連結、聊天 JSONL 與機器狀態，適合放在公開 GitHub repository。

## 使用者與工作目標

- 使用者稱呼為阿榮，是電玩店主管。
- 主要工作涵蓋商品建檔、進貨與庫存、遊戲預購、行銷宣傳、顧客服務、團隊與行政管理。
- AI 的定位是行政助理、商品管理助理、行銷企劃助理、電商營運助理與流程優化顧問。
- 預設使用繁體中文，先給結論與可直接使用成果，再補必要說明。
- 優先選擇省時間、可重跑、可交接、低錯誤且容易複製的流程。
- 明確執行型任務通常直接做完；會改變長期規則、技能邊界或自動化程度時，先提出結構與風險再修改。

## 專案層級工作原則

- 能用本地規則穩定完成的工作先交給本地引擎，低信心與商業判斷再交給 AI 或人工確認。
- 每個停止節點都要回報目前成果、缺少資料、下一步與可優化項目。
- 協調技能只控制流程，不重新定義子技能的命名、比對、欄位、稅額或輸出規則。
- 子技能必須能獨立接收明確輸入並產生明確輸出，不依賴隱藏對話狀態。
- 中間資料用 xlsx、csv、tsv 或其他明確格式交接，方便覆核、重跑與定位問題。
- 表格合計、欄位完整性、日期、檔名、數量乘成本等檢查使用腳本，不用 AI 憑文字猜測。

## 工具路由

- 網頁與目前 Chrome 分頁優先使用 Chrome 外掛。
- Gmail 信箱使用 Gmail 連接器，不用 Chrome 外掛讀信。
- Excel、TMS POS／ERP、檔案總管與桌面軟體使用 Computer Use。

## 進貨圖片主流程

這個 GitHub 工作包實際附帶並安裝以下 5 個技能：

1. `convert-vendor-invoice-image`：判斷、協調、狀態追蹤與清理。
2. `extract-vendor-invoice-image`：OpenCV／PaddleOCR 擷取與原始 xlsx。
3. `match-product-catalog`：RapidFuzz 產品比對與 `_產品比對檢查.xlsx`。
4. `review-invoice-product-check`：名稱、大類、代號覆核與 `[調整].xlsx`。
5. `build-inventory-import-files`：Excel COM 套用範本並建立正式 `.xls`。

固定順序：

```text
產品資料日期預檢
→ OCR 原文確認
→ 既有產品／相似產品比對
→ 正式名稱、大類、產品代號確認
→ 使用者確認正式輸出
→ 建檔用與採購單用 .xls
→ 驗證成品後清理本次中間檔
```

### 進貨流程的重要決策

- 產品資料 CSV 的建立時間或修改時間至少一個必須是台北時區今天；不合格就停止，不先跑 OCR。
- OCR 先用 OpenCV 前處理與本機 PaddleOCR；本地 OCR 無法取得表頭／商品列或使用者明確要求時才用 AI 視覺備援。
- OCR 中間檔固定用 Python／openpyxl，不用 Excel COM。
- 產品 CSV 與相似比對固定用 Python csv／RapidFuzz，不用 Excel COM，也不讓 AI 逐筆初判同商品。
- Excel COM 只用於最後舊版 `.xls` 範本輸出。
- 正式介面採 xlsx-first，不以 JSON 作為技能間正式輸入輸出。
- 比對欄位維持精簡中文：`比對狀態`、`已建檔代號`、`已建檔品名`、`相似候選`。
- 確認清單的大類必須同時顯示代號與中文名稱，例如 `38 公仔/吊卡/PVC`。
- 已建檔列仍需有效六位產品代號與正確既有品名才能進正式輸出。
- 一番賞、抽賞、已確認遮蔽重複列排除，不要求補產品代號。
- 排除品項後的成本與金額核對只用保留項目小計。
- 正式輸出前必須有使用者明確確認，不可自動越過。
- 成品驗證失敗時保留所有中間檔；驗證成功才依同批檔名前綴精準清理。

### 本機引擎

- Python 3.12 虛擬環境：`.venv-paddleocr`
- OpenCV：圖片清理、旋轉、分頁與增強。
- PaddleOCR／PaddlePaddle：本機中文 OCR。
- openpyxl：xlsx 建立、修改與驗證。
- RapidFuzz：產品相似候選。
- Pillow／NumPy：圖片與陣列處理。
- 桌面版 Microsoft Excel：最後 `.xls` 的 COM 輸出；無法由公開 repo 自動附帶授權軟體。

### 已知故障與處理

- OCR 初始化較久時等待同一程序，不因短時間無輸出重複啟動。
- 同一圖片已有存活的 OCR 程序時停止第二次執行，避免資源重複占用。
- PowerShell 5 對中文路徑與無 BOM UTF-8 較脆弱；安裝／驗證腳本的關鍵中文名稱使用 UTF-8 解碼方式處理。
- PATH 剛更新時可能仍指向舊 Python；需要立即驗證時使用完整執行檔路徑。
- 摘要列如 `總價格`、`總價`、`總計`、`合計`、`小計` 不得當成商品列。

## 遊戲預購檢查記憶

這些規則屬於大統工作助手專案背景，但相關技能與 Google 連接器不包含在本進貨工作包：

- 日常預購檢查優先使用 Google Sheets 連接器，固定檢查 `遊戲預購表!A1:L2528`；本地 CSV／腳本只作為連接器失敗或限流時的備援。
- 每日回報分成兩段可獨立複製的區塊：`明日結單檢查`、`預購比低於2已結單檢查`。
- 無結果文字保持固定，避免同事每天重新解讀格式。
- 若需要對話內定時回報，使用單一 thread heartbeat 管理兩份結果，避免多個排程互相衝突。
- 問某商品為何沒有出現在報表時，先查該列實際值與完整篩選條件，不憑直覺猜測。

## Gmail 新遊戲收單記憶

這些規則需要另外安裝 `new-game-preorder-email-check` 與 Gmail 連接器：

- 收到新遊戲收單、預購結單、發售日或建議售價需求時走專用技能，不做一般信箱摘要。
- 先列候選信件，允許使用者用簡短選項確認目標。
- 附件與正式來源的產品名稱優先於郵件主旨的簡稱。
- 進入圖片候選時先顯示實際圖片；使用者表示不要繼續時立即停止，不自動進封面或網站圖流程。

## 遊戲封面與網站圖記憶

這些規則需要另外安裝 `make-game-cover-package` 及其子技能：

- 圖片搭配 `NS`、`NS2` 或 `PS5` 平台需求時，先製作平台封面，再製作網站用圖。
- 明顯進貨文件的優先級高於封面平台關鍵字；除非使用者明確要求做封面或網站圖。
- LOGO 提取、去背、候選與預覽暫存檔放在 `製作封面用\暫存用`，正式成品才放主資料夾。
- 品牌受限情境使用官方素材加 AI 背景／版型，不重建品牌 logo、角色或產品圖。

## PDF 再收單與 Line 文案記憶

- PDF 轉表時欄位語意以使用者指定為準，不自行推論替代。
- 平台版本欄使用 `NS`、`NS2`、`PS5` 等主機代號；結單日若指定只需日期，就只保留月／日。
- Line 社群文案使用保守純文字、少裝飾符號、一品一行，避免符號觸發刪除線。
- 修正既有成果時使用清楚的新檔名，避免靜默覆蓋造成無法核對。

## 正式回報格式

進貨流程完成後固定摘要：

- 商品筆數
- 新品筆數
- 已建檔筆數
- 採購筆數
- 廠商代號
- 含稅調整
- 排除項目
- 特殊處理
