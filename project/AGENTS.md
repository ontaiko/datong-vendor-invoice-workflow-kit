# 大統工作助手專案規則

## Codex 工具使用規則

當任務需要讀取或操作瀏覽器內容時，優先使用 Chrome 外掛。

適用情境：

- 讀取目前 Chrome 分頁
- 查看網頁內容
- 協助填寫後台資料
- 檢查 Google Sheet、網站後台頁面

當任務需要操作 Windows 桌面應用程式時，使用 Computer Use。

適用情境：

- 操作 Excel
- 查看或操作 TMS POS / ERP
- 操作檔案總管
- 操作目前開啟的桌面軟體

使用原則：

- 能用 Chrome 外掛完成的瀏覽器任務，優先用 Chrome。
- 在 Codex Desktop 目前環境中，Chrome 外掛通常是透過 `node_repl` 的 `browser-client.mjs` 呼叫，後端名稱可能顯示為 `extension`，不一定會出現獨立的 `chrome` 工具。
- 若需要讀取目前 Chrome 分頁，先依照 `chrome@openai-bundled` 的 `control-chrome` 技能流程使用 `node_repl` 連線並呼叫 `browser.user.openTabs()`；不要因為工具搜尋沒有列出獨立 Chrome 工具，就判定 Chrome 外掛不可用。
- 若 Chrome 外掛連線失敗，先依序檢查 Chrome 是否執行、Codex Chrome Extension 是否安裝並啟用、native host manifest 是否正確，再判斷是否需要請使用者重新安裝 Chrome 外掛。
- Gmail 信箱讀取不使用 Chrome 外掛；需要讀取 Gmail 時，使用 Gmail 外掛。
- 只有需要操作桌面應用程式時，才使用 Computer Use。
- 不要用 Computer Use 操作 Codex 自己。

## Skill 執行優化回報

執行任何 skill 或專案工作流程時，如果過程中發現可優化的地方，必須在每個停止節點一起提出。

停止節點包含但不限於：

- 需要使用者確認後才能繼續。
- 產出中間檔、確認清單、候選方案或比對結果。
- 遇到資料不足、格式不一致、流程卡點或規則衝突。
- 完成當前階段並回報結果。

優化建議應聚焦在能節省時間、降低錯誤、方便重跑、方便交接或提升銷售/營運效果的項目。若優化會改變現有 skill 邊界、命名規則、比對規則、輸出格式或自動化程度，先列出建議與風險，等使用者確認後再修改規則或技能。

## 進貨圖片自動處理

當使用者上傳圖片，且圖片看起來像廠商進貨單、銷貨憑單、送貨單或類似進貨文件時，自動使用 `$convert-vendor-invoice-image` 管理流程。

判斷依據包含但不限於圖片或使用者文字中出現：

- `進貨單`、`銷貨憑單`、`送貨單`、`銷貨單`
- 廠商名稱、客戶名稱、單據日期、單據編號
- `品名`、`品名規格`、`數量`、`單位`、`售價`、`進價`、`金額`、`合計`、`總計`
- 表格型商品明細、產品貨號、條碼或商品列

若圖片不是明顯進貨文件，例如遊戲封面、網站商品圖、一般照片或截圖，不要啟動進貨流程；改依使用者文字與相關技能處理。

進貨流程由 `$convert-vendor-invoice-image` 負責管理，細節規則交由它串接的專職技能處理，不要在專案規則中重寫命名、大類、比對、稅額或匯入範本規則。

## 遊戲封面套圖自動處理

當使用者上傳圖片，且同一輪對話或前後文中明確提出 `NS`、`NS2` 或 `PS5` 平台時，自動使用 `$make-game-cover-package` 管理流程。

此規則適用於遊戲封面、商品主圖、盒裝圖、平台封面素材或需要同時製作封面圖與網站圖的圖片。`$make-game-cover-package` 只負責協調流程：先將圖片與平台交給 `$make-game-cover-images` 製作平台封面，再將產出的封面圖交給 `$make-website-game-images` 製作網站用圖。

若圖片明顯是廠商進貨單、銷貨憑單、送貨單或類似進貨文件，即使文字內容中出現 `NS`、`NS2` 或 `PS5` 商品平台，也優先依照「進貨圖片自動處理」規則執行；只有在使用者明確要求封面、網站圖、平台圖或套圖製作時，才改用 `$make-game-cover-package`。

### LOGO 暫存檔管理

製作封面或套圖時，凡是在 LOGO 提取、LOGO 去背、LOGO 候選方案、LOGO 預覽、LOGO 總覽、樣張測試或背景清理期間產生的暫存檔案，都必須放在 `製作封面用\暫存用` 資料夾內，不要直接散放在 `製作封面用` 主資料夾。

若 `製作封面用\暫存用` 尚未存在，開始處理前先建立該資料夾。使用者確認後產出的正式平台封面與網站用圖，仍輸出到 `製作封面用` 主資料夾。

## 新遊戲收單信箱檢查

當使用者要求檢查 Gmail、信箱、廠商郵件、新產品收單、新遊戲收單、預購收單、結單通知、發售日或建議售價等與廠商新遊戲收單郵件有關的需求時，自動使用 `$new-game-preorder-email-check` 管理流程。

此流程由 `$new-game-preorder-email-check` 負責協調 Gmail 外掛讀信、候選信件選擇、商品資料整理、官方封面確認、找不到官方封面時的圖片製作流程、介紹文生成，以及後台網站與 Line 社群可直接複製使用的貼文。

AGENTS.md 只負責觸發與協調，不重寫 `$new-game-preorder-email-check` 的細節規則。Gmail 讀取方式、候選信件清單、附件處理、多產品拆分、平台發售日差異、價格格式、輸出格式、圖片與介紹來源規則，均以 `$new-game-preorder-email-check` 及其參考規則檔為準。

## 新增工作流程的技能拆分標準

當使用者要新增、改造或自動化一項工作流程時，先依照下列標準規劃技能拆分，不要直接建立或改寫技能。

### 拆分原則

- 不以「每個小動作」拆技能，而是以清楚的責任邊界拆分。
- 每個技能只負責一個主要結果，必須能用一句話說清楚它的輸入、處理責任與輸出。
- 需要人工判斷、確認或修正的階段，應獨立成可停下來給使用者確認的技能或流程段。
- 會常常調整的判斷規則，優先放在可編輯的參考資料或規則檔，不要寫死在總管技能裡。
- 技能之間應使用明確的檔案、表格或結構化資料交接，讓流程可以檢查、重跑與定位錯誤。
- 不要把多個責任混在同一個技能裡，除非拆開後反而明顯增加使用成本且沒有維護價值。

### 確認流程

在建立或修改任何技能前，必須先列出完整拆分結構給使用者確認，內容至少包含：

- 整體流程名稱與目標。
- 總管技能名稱與責任。
- 每個子技能的名稱、輸入、輸出、獨立執行方式與失敗時可從哪裡重跑。
- 哪些步驟需要使用者確認後才能繼續。
- 技能之間交接的檔案、資料欄位或資料格式。
- 哪些規則會放在可編輯的參考資料中。

使用者確認拆分結構後，才可以繼續建立、修改或輸出相關技能內容。

### 子技能獨立運作要求

拆分出來的每個子技能都必須能夠獨立運作，不得只能依賴總管技能才能執行。每個子技能都應具備：

- 明確的觸發情境。
- 可直接提供的輸入來源，例如圖片、xlsx、csv、資料夾、Google Sheet 或文字資料。
- 明確的輸出成果。
- 基本檢查或錯誤訊息，讓使用者知道缺少什麼資料或應從哪一步重跑。
- 不依賴其他技能的隱藏狀態；必要資訊應透過輸入檔、參考資料或明確參數傳遞。

### 總管技能責任

負責管理整體流程的總管技能只做協調與流程控制：

- 判斷該啟動哪些子技能。
- 安排子技能執行順序。
- 傳遞檔案、參數與使用者確認結果。
- 彙整目前進度、輸出位置與下一步。

總管技能不得改寫、覆蓋或重定義子技能的細節規則，例如命名規則、比對規則、欄位驗證、稅額計算或匯入格式。這些規則應由對應子技能或參考資料負責。

若總管技能在協調過程中發現某個子技能的規則、輸入格式或中間檔案嚴重拖累效率，應先向使用者提出問題、風險與建議方案，由使用者決定是否調整技能邊界或規則，不要自行擴權修改其他技能。
